# 城市出租车出行数据分析与智能问答系统 —— 人机协作报告

## 一、 AI 交互日志

在本次期末大作业的开发过程中，我广泛使用了 AI 工具来辅助代码生成和错误排查。整体而言，AI 极大地缩短了 API 查阅和基础逻辑编写的时间，但在项目架构、本地环境适配和库版本更迭上，依然需要大量的人工干预与纠偏。以下是按模块记录的核心交互过程：

### 1\. M1 数据处理模块

* **关键 Prompt**：“我需要处理 2026 年 1 月的纽约出租车 parquet 数据。请帮我用 pandas 写一个数据质量报告函数，包含缺失率、唯一值，并统计数值型特征中超出 3 倍标准差的异常值比例。另外，请帮我设计两个有意义的衍生特征。”
* **AI 输出摘要**：AI 给出了加载 parquet 文件的代码，并利用 3-Sigma 准则编写了异常值检测函数。在衍生特征方面，AI 建议了“行程耗时”和“平均行驶速度”。
* **采用/修改/拒绝决策**：**修改并人工修正**。
* **AI 犯错案例分析**：

  * **错误表现**：在生成异常值报告时，AI 给出的代码是 `for col in df.select\\\_params(include=\\\[np.number]).columns:`。运行后直接抛出 `AttributeError: 'DataFrame' object has no attribute 'select\\\_params'`。
  * **如何发现**：在终端执行 `python main.py` 时，程序在 M1 阶段直接崩溃中断。
  * **如何修正**：这属于 AI 典型的 API 幻觉，将其他库的用法混淆到了 Pandas 中。我手动将其修正为 `select\\\_dtypes`，解决了报错。此外，在计算“平均行驶速度”时，AI 直接用距离除以了 `timedelta` 时间对象，导致类型冲突。我将其修正为 `.dt.total\\\_seconds() / 60.0`，计算出正确的分钟数。

### 2\. M2 分析可视化模块

* **关键 Prompt**：“使用 seaborn 绘制上客量最高的 TOP 10 区域柱状图。另外，使用 geopandas 加载 taxi\_zones.shp 文件，将刚才的统计结果映射到地图上，绘制区域分级设色地图。”
* **AI 输出摘要**：AI 提供了标准的 Matplotlib 和 Seaborn 绘图代码 `sns.barplot(x=..., y=..., palette='viridis')`，并给出了 Geopandas 加载 shapefile 文件并使用 `merge` 函数进行左连接的代码。
* **采用/修改/拒绝决策**：**修改**。
* **AI 犯错案例分析**：

  * **错误表现**：

    1. 控制台弹出大量警告：`FutureWarning: Passing palette without assigning hue is deprecated...`。
    2. Geopandas 抛出严重报错：`pyogrio.errors.DataSourceError: Unable to open data/taxi\\\_zones.shx`。
  * **如何发现**：代码执行到地图渲染阶段时直接崩溃。
  * **如何修正**：第一个问题说明 AI 的知识库未适配最新版 Seaborn 的严格规范。我手动为 `sns.barplot` 增加了 `hue` 参数，并设置 `legend=False` 清除了警告。第二个问题是因为 AI 假定 shapefile 相关文件全都在 `data/` 根目录下，而实际解压后多了一层 `taxi\\\_zones/` 子文件夹。我将路径显式修改为 `data/taxi\\\_zones/taxi\\\_zones.shp`，确保了底层的 `.shx` 和 `.dbf` 索引文件能被正确加载。

### 3\. M3 预测模型模块

* **关键 Prompt**：“请用 PyTorch 构建一个神经网络，用随机森林做对比，预测某区域某时段的出行需求量。按 8:2 划分数据集，固定随机种子，并绘制 loss 曲线保存到 outputs 目录。”
* **AI 输出摘要**：AI 编写了完整的数据聚合逻辑（groupby 区域、时间、周末等特征），使用 `StandardScaler` 进行特征标准化，搭建了一个两层隐藏层的全连接网络，并自动生成了对比指标的 CSV 文件。
* **采用/修改/拒绝决策**：**部分重构**。
* **AI 犯错案例分析**：

  * **错误表现**：AI 提供的代码在单模块内运行完美，但缺乏全局架构思维。它将模型训练、评估和数据打印全部封装在一个没有返回值的函数里。
  * **如何发现**：在后续开发 M4 问答系统，试图让用户通过指令调用预测模型时，发现 M3 根本没有向外暴露推理接口，也没有把训练好的模型对象传出来。
  * **如何修正**：我重构了 M3 的业务逻辑，将其返回值修改为 `return agg\\\_df, rf\\\_model`，并单独抽离出了一个 `predict\\\_demand(model, location\\\_id, hour, weekday)` 接口，彻底打通了 M3 与 M4 之间的数据流。

### 4\. M4 问答接口模块

* **关键 Prompt**：“我想完成 M4 的可选加分项，将当前的命令行问答循环用 Gradio 封装成一个网页 UI。同时，需要优化正则匹配，能够动态提取用户句子里的参数，比如提问‘周一10点区域237的需求’，系统要能把这三个变量抽出来送入模型。”
* **AI 输出摘要**：AI 使用 `gr.ChatInterface` 构建了前端交互，并给出了利用 `re.search` 分组捕获动态参数的代码。
* **采用/修改/拒绝决策**：**高度采用并调试**。
* **AI 犯错案例分析**：

  * **错误表现**：在调试界面时，直接运行 `m4\\\_qa\\\_system.py` 会抛出 `ModuleNotFoundError: No module named 'src'`。
  * **如何发现**：在 VS Code 中点击运行当前文件时触发。
  * **如何修正**：这暴露了 AI 在处理多文件 Python 工程时的盲区。它生成的相对导入代码 `from src.m3\\\_modeling import ...` 只有在项目根目录运行 `main.py` 时才有效。我确立了严格的运行规范，必须通过主入口启动，确保了项目上下文路径的统一。

## 二、 三阶段代码对比：自然语言意图识别与参数提取

本部分选取 M4 问答模块中的核心功能——**“多条件动态需求预测”**，展示代码从原生编写、到 AI 辅助、再到自然语言对话驱动重构的三阶段演进过程，分析不同阶段的执行效率与理解深度。

### 第一阶段：Native 版（独立硬编码）

在没有任何参考和 AI 辅助时，我采用了最基础的字符串包含判断来处理问答。

```python
def qa\\\_native(query, rf\\\_model):
    # 只能应对格式极其严格的提问，拓展性为零
    if "周一" in query and "10点" in query and "区域237" in query:
        pred = rf\\\_model.predict(\\\[\\\[237, 10, 0, 0]])
        return f"预测结果为: {int(pred\\\[0])}"
    elif "周二" in query and "9点" in query and "区域161" in query:
        pred = rf\\\_model.predict(\\\[\\\[161, 9, 1, 1]])
        return f"预测结果为: {int(pred\\\[0])}"
    else:
        return "无法识别的预测条件，请重新输入。"
```

* **效率与理解深度**：开发效率极低。这种写法属于典型的“穷举法”，一旦用户的区域 ID 或时间发生变化，代码就彻底失效了。它仅仅是在做死板的文本比对，完全不具备任何“智能”解析的能力，在面对真实的测试用例时毫无用处。

### 第二阶段：Prompt 版（AI 辅助引入正则）

向 AI 提出需求后，AI 建议引入正则表达式来提取字符串中的结构化数据。

```python
def qa\\\_prompt(query, rf\\\_model):
    import re
    # 抽象出星期、小时、区域三个实体
    match = re.search(r'周(.)(\\\\d+)点.\\\*区域(\\\\d+)', query)
    if match:
        day\\\_map = {'一':0, '二':1, '三':2, '四':3, '五':4, '六':5, '日':6}
        weekday\\\_str = match.group(1)
        hour = int(match.group(2))
        zone\\\_id = int(match.group(3))
        
        if weekday\\\_str in day\\\_map:
            weekday = day\\\_map\\\[weekday\\\_str]
            pred = rf\\\_model.predict(\\\[\\\[zone\\\_id, hour, weekday, 0]])
            return f"该时段预测需求量: {int(pred\\\[0])} 单"
    return "解析失败，请检查输入格式。"
```

* **效率与理解深度**：代码复用率大幅提升。通过 AI 提供的 `re.search` 搭配括号捕获组的方案，系统具备了解析动态数据的能力。此时的代码已经能处理几十种不同的组合查询。但它的缺点在于缺乏容错和业务解释深度，一旦匹配失败直接返回生硬的提示，且 `is\\\_peak` 这种需要根据时间动态推导的特征被 AI 写死了 `0`，预测精度受到影响。

### 第三阶段：Vibe 版（自然语言对话驱动与工程化重构）

在明确了业务意图后，我与 AI 进行了深入对话，指出“预测时需要根据周末和小时动态判断高峰期”、“需要把干瘪的数字 ID 翻译成具体的纽约地名”，并要求将输出格式重构为 Markdown 渲染的 Web 流。

```python
def qa\\\_vibe(query, rf\\\_model, zone\\\_mapping):
    # 模糊意图匹配 + 动态参数捕获的工业级写法
    if re.search(r'预测|需求|多少单', query) and re.search(r'区域', query):
        day\\\_match = re.search(r'周(一|二|三|四|五|六|日)', query)
        hour\\\_match = re.search(r'(\\\\d+)\\\[点时]', query)
        zone\\\_match = re.search(r'区域(\\\\d+)', query)
        
        # 严格的条件完整性校验
        if day\\\_match and hour\\\_match and zone\\\_match:
            day\\\_map = {'一':0, '二':1, '三':2, '四':3, '五':4, '六':5, '日':6}
            weekday\\\_str = day\\\_match.group(1)
            weekday = day\\\_map\\\[weekday\\\_str]
            hour = int(hour\\\_match.group(1))
            zone\\\_id = int(zone\\\_match.group(1))
            
            # 动态判断是否为高峰期
            is\\\_peak = 1 if weekday < 5 and (hour in \\\[7, 8, 9, 16, 17, 18, 19]) else 0
            
            # 调用封装好的预测层
            pred\\\_demand = predict\\\_demand(rf\\\_model, zone\\\_id, hour, weekday, is\\\_peak)
            zone\\\_name = zone\\\_mapping.get(zone\\\_id, '未知区域')
            
            # Markdown 富文本输出
            return f"\\\*\\\*\\\[结论]\\\*\\\*: 预测需求量约为 {pred\\\_demand} 单/小时\\\\n\\\\n\\\*\\\*\\\[解释]\\\*\\\*: 基于周{weekday\\\_str}的 {hour} 点，区域 {zone\\\_id} ({zone\\\_name}) 的预测打车需求为 {pred\\\_demand} 单。\\\\n\\\\n\\\*\\\*\\\[文件路径]\\\*\\\*: outputs/m3\\\_model\\\_metrics.csv"
        else:
            return "触发了【需求预测】功能，但信息不全。请提供：'预测周一10点区域237的需求'"
```

* **效率与理解深度**：这一版的代码达到了极高的健壮性。通过与 AI 反复讨论业务逻辑，代码解耦了模型推理、字典映射和异常处理。AI 在此阶段不仅仅是生成代码，而是在配合落地系统设计，处理了动态特征 `is\\\_peak` 的回填，并使用 Gradio 的 Markdown 引擎实现了图文并茂的流式输出。

## 三、 反思：对 AI 工具能力边界的全新认识

完成本次期末大作业后，经历了从数据处理、模型训练到网页端交互部署的全流程，我对 AI 的实际工作能力有了更直观的体会：

首先，AI 编写独立代码块的速度极快，能在短时间内给出复杂的 PyTorch 网络结构和 Pandas 逻辑。但在模块串联上，它缺乏全局视角。它不会主动意识到 M3 的模型在训练完成后需要保留在内存中，以便传递给 M4 模块调用。如果没有人工进行自上而下的模块化设计和数据流规划，它生成的代码往往无法互相通信，导致工程断裂。

其次，AI 在处理底层环境和版本迭代时存在盲区。在处理 M2 的地图数据时，AI 给出的代码逻辑看似合理，但实际运行中由于 Seaborn 参数规则更新以及 Geopandas 对子目录路径的解析问题，直接导致了报错。AI 的知识库是静态的，无法实时感知本地真实的文件树结构或第三方库的版本变动，这要求我在调试环境和查阅官方文档上投入大量精力。

最后，AI 的输出质量高度依赖于输入指令的精确度。在开发 M4 问答模块时，AI 初始提供的正则匹配逻辑过于单一。只有在详细界定业务场景——说明用户提问的随机性，指出需要动态抓取数字和时间参数后——它才输出了具备泛化能力的匹配逻辑。这表明，AI 工具的能力上限取决于使用者拆解问题和把控全局逻辑的能力。理清业务逻辑，才能引导 AI 生成贴合实际需求的系统代码。
